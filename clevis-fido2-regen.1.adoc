CLEVIS-FIDO2-REGEN(1)
======================
:doctype: manpage


== NAME

clevis-fido2-regen - Rotates the hmac-salt of an existing clevis+fido2 LUKS2 binding.

== SYNOPSIS

*clevis-fido2-regen* -d DEV [-s SLT] [-y]

== OVERVIEW

The *clevis-fido2-regen* command rotates an existing *clevis encrypt fido2* LUKS2
binding: it generates a fresh, random hmac-salt (and therefore a fresh wrapping key)
while reusing the same enrolled FIDO2 credential, so no new credential creation /
re-enrollment is required.

This tool exists because `clevis luks regen` does not work for third-party pins such
as "fido2" -- it only knows how to reconstruct the configuration for a small set of
built-in pins. Instead, `clevis-fido2-regen` reads the existing binding's own metadata
directly and drives clevis's public `clevis luks bind`/`clevis luks unbind` commands.

It only supports LUKS2 devices.

== HOW IT WORKS

1. The existing fido2 binding's configuration (type, rp_id, cred_id, up, uv, pin,
   resident, device, timeout, aaguid) is read directly from the LUKS2 token/JWE
   metadata for the target slot.
2. A LUKS header backup is taken (path printed to stderr) before anything is changed.
3. A *new* keyslot is added via `clevis luks bind`, reusing the same "cred_id" (so the
   token will ask for user presence/PIN, but will not need to create a new credential),
   with a freshly generated hmac-salt.
4. The new slot is independently verified to actually decrypt and unlock the device.
5. Only after that verification succeeds is the old slot removed via
   `clevis luks unbind`.

Expect to touch/verify your token up to three times: once to unlock the existing
binding (to recover the LUKS key), once for the new binding's assertion, and once
more to verify the new binding actually works before the old slot is removed.

If verification fails at any point, the old slot is left untouched and the (new,
unverified) slot is reported so it can be inspected or removed manually.

== OPTIONS

* *-d DEV* :
  The LUKS2 device containing the fido2 binding to rotate. Required.

* *-s SLT* :
  The LUKS2 keyslot to rotate. If omitted, it is auto-detected -- this only works
  if the device has exactly one clevis+fido2 binding; otherwise you must specify
  which slot explicitly.

* *-y* :
  Skip the confirmation prompt.

== EXAMPLES

    # clevis-fido2-regen -d /dev/sda2
    Regenerating clevis+fido2 binding on /dev/sda2, slot 1:
      rp_id=fido2.clevis cred_id=AAAA... type=es256
    This will add a new slot bound with a fresh hmac-salt (same credential),
    verify it works, then remove slot 1. It will touch your token up to 3 times.
    Do you want to proceed? [yN] y
    LUKS header backed up to /root/clevis-fido2-regen-sda2-20260719120000.header-backup -- keep this until you've confirmed the device unlocks normally.
    New slot 2 added. Verifying it actually unlocks the device (one more touch/PIN)...
    New slot 2 verified working. Removing old slot 1...
    Done: /dev/sda2 slot 1 rotated to slot 2. Header backup kept at /root/clevis-fido2-regen-sda2-20260719120000.header-backup.

== CAVEATS

This tool has not been exercised against real FIDO2 hardware. Dry-run it on a
throwaway LUKS2 volume before relying on it for anything real, and keep the header
backup it prints until you have confirmed the device unlocks normally on next boot.
